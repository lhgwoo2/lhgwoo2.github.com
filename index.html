    기술 영역

    센치한 아침에는 이런 기술들이 사용되었어요!

    Firebase Realtime Database

    날씨 태그의 곡 정보를 realtime DB에 넣어 언제든지 곡 정보를 업데이트 또는 수정할 수 있도록 합니다.

    Firebase Cloud Storage

    알람이 울릴 때 표시될, 날씨 배경이 저장되어 있습니다.

    Glide

    앱 내의 이미지들을 로딩해줄 이미지 로딩 라이브러리 Glide가 사용되었습니다
    컨번스뷰나 또는 재생목록에서 앨범이미지를 표시할 때 활용됩니다.

    Jamendo Music API

    알람이 울릴 때 재생될 노래를 가져오는 음악 스트링밍 API입니다.

    Open Weather Map API

    알람이 울리는 장소의 날씨를 가져올 떄 사용되는 날씨 API입니다.
    해당 날씨를 통하여 cloud에서 날씨에 맞는 배경화면을 가져옵니다.

    Realm Local Database

    등록된 알람과 재생된 노래의 정보를 저장하는 로컬 저장소로 Realm DB가 사용되었습니다.

    AsyncTask 병렬처리

    알람이 호출될 때 여러 데이터를 동시에 호출해야 했습니다.

    AsyncTask를 활용하면 되겠지 막연히 생각하였던 것이 Android 3.0 이상의 버전에서는 순차적으로 실행되는 것을 뒤늦게 깨달았습니다.

    그리하여 좀더 빠른 처리를 위해 병렬적으로 스레드를 활용해야 했고, executeOnExecutor()메서드를 활용하면 AsyncTask를 병렬적으로 동작시킬 수 있다는 것을 발견하였다.

    이것은 스레드가 병렬적으로 동작하여 빠른 데이터 처리를 가능케 하였습니다.



    Fragment가 생성되고 Manager에서 관리될 때, 항상 같은 스택순서로 프래그먼트가 쌓이지 않는 것을 발견하였습니다.

    처음 테스트를 할 때는, 작동 상 오류가 없었기에 발견하지 못하다가 가끔 한 번씩 앱이 다운되면서 오류를 뒤늦게 발견하였습니다.

    그러나 가끔씩 발생하는 오류였기에, 버그를 찾기 어려웠습니다.

    그러던 중 팀원의 도움으로 디버그 모드를 처음으로 사용해보게 되었고, Fragment Manager 객체 내부를 살펴보던 중 다른 스택 순서로 Fragment가 쌓이는 것을 발견하여 오류를 해결할 수 있었습니다.

    이미지 사이즈 관리

    알람화면에서 배경화면을 띄워주어야 하기 때문에 큰 사이즈의 이미지가 사용되었습니다.

    그렇기 때문에 용량이 상대적으로 클 수 밖에 없는데, 처음에는 이미지를 Glide를 활용하여 캐싱된 이미지를 뷰에 그대로 띄워주었습니다.

    그러나 몇몇 경우에는 OutOfMemory가 발생하여 이미지 로딩을 실패하였고, 이미지 및 캐시메모리를 앱에 최적화 해야함을 깨달았습니다.

    먼저, Heap메모리 사이즈를 변경하였고, 메모리가 부족할 경우 캐싱된 메모리를 비워주는 작업을 하였습니다.

    또한 이미지 품질을 앱에 큰 영향이 끼치지 않는 선까지 낮추어 용량을 최적화 하였습니다.

    그리하여 이미지 로딩 실패를 방지할 수 있었고, 이미지 작업에서는 항상 최적화가 필요하다는 것을 다시 한번 깨달았습니다.



    처음 안드로이드 개발을 해본 입장으로서 1가지 목표가 있었습니다.

    "앱을 완성해서 Play 스토어에 올려 서비스까지 해보자" 였습니다.

    원했던 목표를 거의 달성하여 의미 있고 즐거웠습니다.

    그리고 더욱 좋았던 것은 다른 프로젝트와 다르게 안드로이드 앱 개발은 개인이 상대적으로 쉽게 서비스를 만들어 제공할 수 있다는 것이었습니다.

    이러한 점이 참으로 매력적이었고 "센치한 아침"을 시작으로 향후 플레이스토어에서 클라이언트들과 소통하고 앱을 더욱 개선해가며 일련의 서비스를 경험해 볼 것이 기대 됩니다.
